# Отчет о реализации собственной версии RxJava

Данный отчет описывает архитектуру, механизмы работы и процесс тестирования кастомной библиотеки реактивного программирования, созданной в рамках задания.

## 1. Архитектура реализованной системы

Архитектура библиотеки базируется на классическом паттерне Наблюдатель (Observer), расширенном поддержкой цепочек преобразований (операторов) и асинхронности.

### Базовые компоненты:
*   **`Observer<T>`**: Интерфейс подписчика, который принимает события через методы `onNext`, получает уведомления об ошибках через `onError` и об успешном завершении через `onComplete`.
*   **`Observable<T>`**: Главный абстрактный класс, представляющий реактивный поток. Содержит базовый метод `subscribe()`, статический фабричный метод `create()` и методы-операторы (`map`, `filter`, `flatMap`, `subscribeOn`, `observeOn`). 
*   **`Emitter<T>`**: Интерфейс, предоставляемый разработчику внутри `Observable.create(...)`, через который происходит безопасная отправка событий в поток.
*   **`Disposable`**: Интерфейс для отмены подписки (`dispose()`).

### Принцип работы цепочек:
При вызове методов-операторов (например, `map` или `filter`) исходный объект `Observable` не изменяется (принцип иммутабельности). Вместо этого создается новый объект-обертка (например, `ObservableMap`), который сохраняет ссылку на исходный `Observable`.
При вызове `subscribe()` на последнем элементе цепочки происходит «раскручивание» подписки вверх до первоначального источника (источника данных), а затем события проталкиваются вниз (push-based подход) от источника через все обертки к конечному подписчику. Управление флагом отмены потока (`isDisposed`) осуществляется в единой корневой обертке, чтобы избежать состояния гонки при многопоточном выполнении.

## 2. Управление потоками выполнения (Schedulers)

Библиотека поддерживает переключение контекста выполнения с помощью интерфейса `Scheduler`. Каждый `Scheduler` инкапсулирует механизм планирования задач (чаще всего на базе `ExecutorService`). 

Реализовано три основных вида:
1.  **`IOThreadScheduler`**:
    *   **Принцип работы**: Использует `Executors.newCachedThreadPool()`. Этот пул создает новые потоки по мере необходимости и переиспользует неактивные.
    *   **Область применения**: Оптимален для I/O-операций (работа с сетью, чтение/запись файлов, запросы к БД), где потоки большую часть времени находятся в состоянии ожидания (blocked/waiting).
2.  **`ComputationScheduler`**:
    *   **Принцип работы**: Использует `Executors.newFixedThreadPool(processors)`. Размер пула жестко ограничен количеством доступных ядер процессора (`Runtime.getRuntime().availableProcessors()`).
    *   **Область применения**: Предназначен для ресурсоемких вычислений (математические операции, обработка больших объемов данных). Ограничение количества потоков предотвращает издержки на переключение контекста (context switching).
3.  **`SingleThreadScheduler`**:
    *   **Принцип работы**: Использует `Executors.newSingleThreadExecutor()`. Все задачи выполняются строго последовательно в одном выделенном потоке.
    *   **Область применения**: Идеален для ситуаций, где необходима строгая упорядоченность выполнения задач (sequential execution) или работа с потоконебезопасными структурами.

### Операторы маршрутизации:
*   **`subscribeOn(Scheduler)`**: Влияет на то, в каком потоке будет вызвана логика самого источника данных (то есть блок кода внутри `Observable.create`).
*   **`observeOn(Scheduler)`**: Меняет поток, в котором будут обрабатываться все последующие события (вниз по цепочке) через вызовы `onNext`, `onError`, `onComplete`.

## 3. Процесс тестирования

Для проверки работоспособности библиотеки был использован фреймворк JUnit 5. Реализован специальный вспомогательный класс `TestObserver`, который перехватывает события и использует `CountDownLatch` для безопасного ожидания завершения асинхронных операций.

**Основные проверенные сценарии:**
1.  **Базовое создание потоков (`testCreateAndSubscribe`)**: Проверка корректности передачи событий от эмиттера к подписчику.
2.  **Операторы трансформации (`testMapOperator`, `testFilterOperator`, `testFlatMapOperator`)**: 
    *   `map` успешно преобразует типы (например, строку в число).
    *   `filter` корректно отбрасывает элементы, не соответствующие предикату.
    *   `flatMap` успешно разворачивает вложенные `Observable` в единый плоский поток событий.
3.  **Обработка ошибок (`testErrorHandling`)**: Проверено, что при вызове `onError` поток немедленно прекращает передачу новых событий и пробрасывает ошибку вниз по цепочке.
4.  **Отмена подписки (`testDispose`)**: Убедились, что при вызове метода `dispose()` обработка новых событий прекращается, а флаг отмены изменяется корректно.
5.  **Многопоточность (`SchedulersTest`)**:
    *   Проверено, что `subscribeOn` переносит создание данных в пул IO.
    *   Проверено, что `observeOn` успешно переносит обработку (маппинг) в пул Computation.
    *   Протестировано корректное совместное использование `subscribeOn` и `observeOn`, а также отсутствие состояния гонки при доставке терминальных событий (`onComplete`/`onError`).

## 4. Примеры использования

### Пример 1. Базовая цепочка обработки
```java
Observable.<String>create(emitter -> {
    emitter.onNext("10");
    emitter.onNext("20");
    emitter.onNext("invalid_number");
    emitter.onComplete();
})
.map(Integer::parseInt) // Преобразуем String в Integer (упадет на "invalid_number")
.filter(num -> num > 15) // Оставляем только числа больше 15
.subscribe(new Observer<Integer>() {
    @Override
    public void onNext(Integer item) {
        System.out.println("Получено: " + item);
    }

    @Override
    public void onError(Throwable t) {
        System.err.println("Ошибка обработки: " + t.getMessage());
    }

    @Override
    public void onComplete() {
        System.out.println("Поток завершен.");
    }
});
```

### Пример 2. Асинхронное выполнение
```java
Observable.<String>create(emitter -> {
    System.out.println("Чтение файла в потоке: " + Thread.currentThread().getName());
    emitter.onNext("file_content_line_1");
    emitter.onComplete();
})
.subscribeOn(Schedulers.io()) // Чтение файла произойдет в пуле IO
.observeOn(Schedulers.computation()) // Обработка результата перейдет в пул Computation
.map(line -> {
    System.out.println("Обработка в потоке: " + Thread.currentThread().getName());
    return line.toUpperCase();
})
.subscribe(new Observer<String>() {
    @Override
    public void onNext(String item) {
        System.out.println("Результат: " + item);
    }

    @Override
    public void onError(Throwable t) { t.printStackTrace(); }

    @Override
    public void onComplete() { }
});
```
