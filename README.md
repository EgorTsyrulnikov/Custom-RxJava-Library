# Отчет по курсовой работе: Custom RxJava Library

В рамках курсовой работы была реализована собственная версия библиотеки реактивного программирования (RxJava), использующая паттерн «Наблюдатель» (Observer pattern) и предоставляющая возможности асинхронного управления потоками выполнения.

## Архитектура и принцип действия

Архитектура библиотеки базируется на классическом паттерне Observer, расширенном поддержкой цепочек преобразований (операторов) и асинхронности.

### Базовые компоненты
*   **`Observer<T>`**: Интерфейс подписчика, который принимает события через метод `onNext`, получает уведомления об ошибках через `onError` и об успешном завершении через `onComplete`.
*   **`Observable<T>`**: Главный абстрактный класс, представляющий реактивный поток. Включает базовый метод `subscribe()`, фабричный метод `create()` и методы-операторы (`map`, `filter`, `flatMap`, `subscribeOn`, `observeOn`). 
*   **`Emitter<T>`**: Интерфейс для безопасной отправки событий в поток, предоставляемый разработчику внутри `Observable.create(...)`.
*   **`Disposable`**: Интерфейс для отмены подписки (`dispose()`).

### Принцип работы цепочек (Operators)
При вызове методов-операторов (например, `map` или `filter`) исходный объект `Observable` не изменяется (принцип иммутабельности). Вместо этого создается новый объект-обертка, который сохраняет ссылку на исходный `Observable`.
При вызове `subscribe()` на последнем элементе цепочки происходит «раскручивание» подписки вверх до первоначального источника (источника данных), а затем события проталкиваются вниз (push-based подход) от источника через все обертки к конечному подписчику. Управление флагом отмены потока (`isDisposed`) осуществляется в единой корневой обертке, чтобы избежать состояния гонки (race conditions) при многопоточном выполнении.

## Управление потоками выполнения (Schedulers)

Библиотека поддерживает переключение контекста выполнения с помощью интерфейса `Scheduler`. Каждый `Scheduler` инкапсулирует механизм планирования задач (на базе пулов потоков). 

### Реализованные пулы
1.  **`IOThreadScheduler`**:
    *   **Механизм**: Использует `Executors.newCachedThreadPool()`. Создает новые потоки по мере необходимости и переиспользует неактивные.
    *   **Область применения**: Оптимален для I/O-операций (работа с сетью, чтение/запись файлов, запросы к БД), где потоки большую часть времени находятся в состоянии ожидания (blocked/waiting).
2.  **`ComputationScheduler`**:
    *   **Механизм**: Использует `Executors.newFixedThreadPool(processors)`. Размер пула жестко ограничен количеством доступных ядер процессора (`Runtime.getRuntime().availableProcessors()`).
    *   **Область применения**: Предназначен для ресурсоемких вычислений. Ограничение количества потоков предотвращает издержки на переключение контекста (context switching).
3.  **`SingleThreadScheduler`**:
    *   **Механизм**: Использует `Executors.newSingleThreadExecutor()`.
    *   **Область применения**: Идеален для ситуаций, где необходима строгая упорядоченность выполнения задач (sequential execution).

### Операторы маршрутизации
*   **`subscribeOn(Scheduler)`**: Влияет на то, в каком потоке будет вызвана логика самого источника данных (то есть блок кода внутри `Observable.create`).
*   **`observeOn(Scheduler)`**: Меняет поток, в котором будут обрабатываться все последующие события (вниз по цепочке) через вызовы `onNext`, `onError`, `onComplete`.

## Процесс тестирования

Для проверки работоспособности библиотеки был использован фреймворк JUnit 5. Реализован специальный вспомогательный класс `TestObserver`, перехватывающий события и использующий `CountDownLatch` для безопасного ожидания завершения асинхронных операций.

**Основные проверенные сценарии:**
1.  **Базовое создание потоков (`testCreateAndSubscribe`)**: Проверка корректности передачи событий от эмиттера к подписчику.
2.  **Операторы трансформации (`testMapOperator`, `testFilterOperator`, `testFlatMapOperator`)**: 
    *   `map` успешно преобразует типы.
    *   `filter` корректно отбрасывает элементы.
    *   `flatMap` успешно разворачивает вложенные `Observable` в единый плоский поток событий.
3.  **Обработка ошибок (`testErrorHandling`)**: Проверено, что при вызове `onError` поток немедленно прекращает передачу новых событий и пробрасывает ошибку вниз по цепочке.
4.  **Отмена подписки (`testDispose`)**: Проверено прекращение обработки новых событий при вызове метода `dispose()`.
5.  **Многопоточность (`SchedulersTest`)**:
    *   Успешный перенос источника в пул IO через `subscribeOn`.
    *   Успешный перенос обработки в пул Computation через `observeOn`.
    *   Протестировано корректное совместное использование маршрутизаторов потоков без состояния гонки при доставке терминальных событий.

## Примеры использования

### Базовая цепочка обработки
```java
Observable.<String>create(emitter -> {
    emitter.onNext("10");
    emitter.onNext("20");
    emitter.onNext("invalid_number");
    emitter.onComplete();
})
.map(Integer::parseInt) // Преобразуем String в Integer (упадет на "invalid_number")
.filter(num -> num > 15) // Оставляем только числа больше 15
.subscribe(new Observer<Integer>() {
    @Override
    public void onNext(Integer item) {
        System.out.println("Получено: " + item);
    }

    @Override
    public void onError(Throwable t) {
        System.err.println("Ошибка обработки: " + t.getMessage());
    }

    @Override
    public void onComplete() {
        System.out.println("Поток завершен.");
    }
});
```

### Асинхронное выполнение
```java
Observable.<String>create(emitter -> {
    System.out.println("Чтение файла в потоке: " + Thread.currentThread().getName());
    emitter.onNext("file_content_line_1");
    emitter.onComplete();
})
.subscribeOn(Schedulers.io()) // Чтение файла произойдет в пуле IO
.observeOn(Schedulers.computation()) // Обработка результата перейдет в пул Computation
.map(line -> {
    System.out.println("Обработка в потоке: " + Thread.currentThread().getName());
    return line.toUpperCase();
})
.subscribe(new Observer<String>() {
    @Override
    public void onNext(String item) {
        System.out.println("Результат: " + item);
    }

    @Override
    public void onError(Throwable t) { t.printStackTrace(); }

    @Override
    public void onComplete() { }
});
```
